do we need virtaul environment in fast api project and what is virtual environment ? explain with example ?
Yes, **you should use a virtual environment for every FastAPI project** (and generally for every Python project).

# What is a Virtual Environment?

A virtual environment is an **isolated Python environment** that has its own:

* Python packages
* Package versions
* Dependencies

It prevents one project's dependencies from affecting another project's dependencies.

---

## Problem Without Virtual Environment

Suppose you have two projects:

### Project A

Requires:

```text
fastapi==0.115.0
```

### Project B

Requires:

```text
fastapi==0.120.0
```

If you install packages globally:

```bash
pip install fastapi==0.115.0
```

Then later:

```bash
pip install fastapi==0.120.0
```

Project A may break because its expected version was replaced.

---

## Solution: Virtual Environment

Create separate environments:

```text
ProjectA/
│
├── venv/
└── app.py

ProjectB/
│
├── venv/
└── app.py
```

Each project has its own package versions.

```text
ProjectA
└── FastAPI 0.115.0

ProjectB
└── FastAPI 0.120.0
```

No conflicts.

---

### Similarity

Both provide **project-level dependency isolation**.

#### Node.js

```text
my-app/
│
├── node_modules/
├── package.json
└── app.js
```

Dependencies are installed inside `node_modules`.

#### Python

```text
my-app/
│
├── venv/

└── main.py
```

Dependencies are installed inside `venv`.

---

## More Accurate Comparison

| Node.js               | Python                |
| --------------------- | --------------------- |
| `node_modules/`       | `venv/`               |
| `package.json`        | `requirements.txt`    |
| `npm install`         | `pip install`         |
| `npm install express` | `pip install fastapi` |

---

## Important Difference

`node_modules` only contains packages.

Example:

```text
node_modules/
├── express
├── axios
└── lodash
```

But a Python virtual environment contains:

1. Packages
2. A copy/link of the Python interpreter
3. Package management tools (`pip`)

Example:

```text
venv/
├── bin/
│   ├── python
│   ├── pip
│   └── activate
│
├── lib/
│   └── site-packages/
│       ├── fastapi
│       └── uvicorn
```

Notice that `venv` even has its own `python` executable.

---

## Better Analogy

A virtual environment is closer to:

* `node_modules`
* plus a project-specific Node runtime

If Node.js had a folder like:

```text
my-app/
│
├── local-node/
│   ├── node
│   ├── npm
│   └── node_modules/
```

that would be much closer to Python's `venv`.

---

## Why Activate a Virtual Environment?

When you run:

```bash
source venv/bin/activate
```

Python changes:

```bash
python
pip
```

to point to the versions inside `venv`.

Example:

Without activation:

```bash
/usr/bin/python
```

After activation:

```bash
/project/venv/bin/python
```

So when you do:

```bash
pip install fastapi
```

it gets installed only inside that project's environment.

---

### Short Answer

If you're coming from Node.js, the easiest mental model is:

> **`venv` is roughly Python's equivalent of `node_modules`, but more powerful because it also contains an isolated Python interpreter and package manager.**

## What `os.getenv()` actually reads

`os.getenv("DB_PASSWORD")` reads from the **system environment** (`os.environ`) — the same place `export DB_PASSWORD=...` in the terminal writes to.

It does **not** open or read `.env` on its own.

```
os.getenv()  →  reads os.environ (system env)
.env file    →  just a text file (Python ignores it unless something loads it)
```

## How `.env` connects to `os.getenv()`

Your `main.py` has:

```3:4:/home/geek-pc/Desktop/vicky/Python/ToDo/main.py
from dotenv import load_dotenv
load_dotenv()             # this will load the environment variables from the .env file and set them in the environment variables of the system
```

`load_dotenv()` reads `.env` and **copies values into `os.environ`**. After that, `os.getenv()` can see them.

```
.env file
   ↓
load_dotenv()  →  puts values into os.environ
   ↓
os.getenv("DB_PASSWORD")  →  reads from os.environ
```

So: **`.env` → `load_dotenv()` → system env → `os.getenv()`**

## Important: import order in your `main.py`

Right now you have:

```python
from routes.toDoRoutes import router   # ← imports database.py, pool created here
from dotenv import load_dotenv
load_dotenv()                          # ← .env loaded too late
```

`toDoRoutes` → `toDoController1` → `database.py` runs **before** `load_dotenv()`, so the pool may be created with defaults instead of `.env` values.

**Fix — call `load_dotenv()` first:**

```python
from dotenv import load_dotenv
load_dotenv()

from fastapi import FastAPI
from routes.toDoRoutes import router as toDoRouter

app = FastAPI()
app.include_router(toDoRouter)
```

## Summary

| Source | Does `os.getenv()` read it directly? |
|--------|-------------------------------------|
| `export` in terminal | Yes |
| `.env` file | Only after `load_dotenv()` |
| Default in code (`"localhost"`) | Used if env var is not set |

**Short answer:** `os.getenv()` reads **system environment variables**. A `.env` file works only if `load_dotenv()` runs **before** `database.py` is imported.

Q-5 :
dedecorators function in python ?


I think you mean **decorator** (`@decorator`) in Python.

A **decorator** is a special function that can **modify or extend the behavior of another function or class without changing its original code**.

---

## Basic Syntax

```python
def my_decorator(func):
    def wrapper():
        print("Before function call")
        func()
        print("After function call")
    return wrapper

@my_decorator
def greet():
    print("Hello!")

greet()
```

### Output

```python
Before function call
Hello!
After function call
```

---

## What does `@my_decorator` mean?

This:

```python
@my_decorator
def greet():
    print("Hello!")
```

is equivalent to:

```python
def greet():
    print("Hello!")

greet = my_decorator(greet)
```

Python passes `greet` as an argument to `my_decorator`, and the returned function replaces the original `greet`.

---

## How It Works

### Step 1: Original Function

```python
def greet():
    print("Hello!")
```

### Step 2: Decorator Receives Function

```python
def my_decorator(func):
    print(func)
```

Here `func` refers to `greet`.

### Step 3: Return Wrapper Function

```python
def my_decorator(func):
    def wrapper():
        print("Before")
        func()
        print("After")
    return wrapper
```

Now whenever `greet()` is called, actually `wrapper()` runs.

---

# Decorator with Arguments

Without `*args` and `**kwargs`, decorators only work for functions with fixed parameters.

```python
def my_decorator(func):
    def wrapper(*args, **kwargs):
        print("Before")
        result = func(*args, **kwargs)
        print("After")
        return result
    return wrapper

@my_decorator
def add(a, b):
    return a + b

print(add(2, 3))
```

Output:

```python
Before
After
5
```

---

# Real-World Example: Logging

```python
def logger(func):
    def wrapper(*args, **kwargs):
        print(f"Calling {func.__name__}")
        return func(*args, **kwargs)
    return wrapper

@logger
def login(username):
    print(f"{username} logged in")

login("vicky")
```

Output:

```python
Calling login
vicky logged in
```

---

# Real-World Example: Authentication

Similar to middleware in Express.js.

### Python

```python
def authenticate(func):
    def wrapper(user):
        if user != "admin":
            print("Access Denied")
            return

        return func(user)
    return wrapper

@authenticate
def dashboard(user):
    print("Welcome to dashboard")

dashboard("admin")
dashboard("guest")
```

Output:

```python
Welcome to dashboard
Access Denied
```

### Express.js Equivalent

```javascript
function authenticate(req, res, next) {
  if (req.user !== "admin") {
    return res.status(401).send("Access Denied");
  }

  next();
}
```

Decorators are often compared to middleware because both wrap existing functionality.

# Built-in Decorators

## 1. `@staticmethod`

```python
class Math:
    @staticmethod
    def add(a, b):
        return a + b
```

Usage:

```python
Math.add(2, 3)
```

No `self` required.

---

## 2. `@classmethod`

```python
class Person:
    count = 0

    @classmethod
    def increment(cls):
        cls.count += 1
```

Here `cls` refers to the class itself.

---

## 3. `@property`

```python
class User:
    def __init__(self, name):
        self._name = name

    @property
    def name(self):
        return self._name
```

Usage:

```python
u = User("Vicky")
print(u.name)
```

Instead of:

```python
u.name()
```

# Decorators in FastAPI

You have already seen decorators in FastAPI:

```python
from fastapi import FastAPI

app = FastAPI()

@app.get("/users")
def get_users():
    return ["Vicky", "John"]
```

Here:

```python
@app.get("/users")
```

is a decorator.

Equivalent to:

```python
def get_users():
    return ["Vicky", "John"]

get_users = app.get("/users")(get_users)
```

The decorator registers the function as a route handler for `/users`.

---

# Python Decorator vs JavaScript

### JavaScript

```javascript
function logger(fn) {
  return function (...args) {
    console.log("Before");
    return fn(...args);
  };
}
```

Usage:

```javascript
const wrapped = logger(add);
```

### Python

```python
@logger
def add(a, b):
    return a + b
```

Python's `@` syntax is just a cleaner way to wrap functions.

---

## Summary

A decorator:

* Takes a function as input.
* Adds extra behavior before/after the function runs.
* Returns a new function.
* Uses `@decorator_name` syntax.
* Commonly used for:

  * Logging
  * Authentication/Authorization
  * Caching
  * Validation
  * Timing execution
  * FastAPI routes (`@app.get`, `@app.post`)
  * Class utilities (`@staticmethod`, `@classmethod`, `@property`)

Think of a decorator as a **wrapper around a function**, similar to how **middleware wraps route handlers in Express.js**.

so do you mean decorator work as call back function in js ?

Yes, that's a good way to think about it initially.

A Python decorator uses the function as an argument, similar to how JavaScript passes a callback to another function.

### JavaScript

```javascript
function logger(callback) {
  return function (...args) {
    console.log("Before");
    callback(...args);
    console.log("After");
  };
}

  console.log("Hello");
}

```

Output:

```text
Before
Hello
After
```

---

### Python Decorator

```python

    def wrapper():
        print("Before")
        func()
        print("After")
    return wrapper

@logger
def greet():
    print("Hello")

greet()
```

Output:

```text
Before
Hello
After
```

---

### What's the difference?

A **callback** is simply a function passed to another function.

```javascript
setTimeout(myFunction, 1000);
```

Here `myFunction` is a callback.

A **decorator** is a specific pattern where:

1. A function is passed as an argument.
2. Another function wraps it.
3. The wrapped version replaces the original function.

In JavaScript, this:

```javascript
greet = logger(greet);
```

is conceptually the same as Python:

```python
greet = logger(greet)
```

The `@logger` syntax is just shorthand for that assignment.

---

### FastAPI Example

When you write:

```python
@app.get("/users")
def get_users():
    return ["user1", "user2"]
```

Python internally does something similar to:

```python
def get_users():
    return ["user1", "user2"]

get_users = app.get("/users")(get_users)
```

The function `get_users` is passed to another function and gets registered as a route.

---

So:

* **Callback** = any function passed to another function.
* **Decorator** = a special pattern that accepts a function, wraps/modifies it, and returns a new function.
* Every decorator uses functions as arguments (callback-like behavior), but not every callback is a decorator.

Q-1 : how bcrypt is working 
every time when encrypt the password always salt will be unique ?
if yes then how it will compare password at time of login  how bcrypt will know that which salt have used for encrypt the password ?

Yes, exactly.

Every time you call:

```python
bcrypt.hashpw(password.encode(), bcrypt.gensalt())
```

a **new random salt** is generated, so even the same password produces a different hash.

Example:

```python
password = "secret123"

hash1 = bcrypt.hashpw(password.encode(), bcrypt.gensalt())
hash2 = bcrypt.hashpw(password.encode(), bcrypt.gensalt())

print(hash1)
print(hash2)
```

Output:

```text
$2b$12$abc...
$2b$12$xyz...
```

The hashes are different even though the password is the same.

---

## How does bcrypt know the salt during login?

The clever part is that **bcrypt stores the salt inside the hash itself**.

A bcrypt hash looks something like:

```text
$2b$12$Kj6x6P9QqR1x0N1j7J8fyuV6R0J8u1Qw5jW0B6jJ6Y6vV8f0F9e
```

Breakdown:

```text
$2b$      -> bcrypt version
12$       -> cost factor (rounds)
Kj6x6P9QqR1x0N1j7J8fyu -> salt
V6R0J8u1Qw5jW0B6jJ6Y6vV8f0F9e -> hashed password
```

So when you save the hash in the database, you're actually saving:

* bcrypt version
* cost factor
* salt
* hash result

all together in one string.

---

## What happens during login?

Suppose the database contains:

```text
$2b$12$Kj6x6P9QqR1x0N1j7J8fyuV6R0J8u1Qw5jW0B6jJ6Y6vV8f0F9e
```

User enters:

```text
secret123
```

Then:

```python
bcrypt.checkpw(
    "secret123".encode(),
    stored_hash.encode()
)
```

Internally bcrypt:

1. Extracts the salt from the stored hash.
2. Extracts the cost factor.
3. Hashes `"secret123"` again using the same salt and cost.
4. Compares the newly generated hash with the stored hash.
5. Returns `True` if they match.

Conceptually:

```python
stored_hash = "$2b$12$Kj6x6P9QqR1x0N1j7J8fyu..."

salt = extract_salt(stored_hash)

new_hash = hash(password_entered, salt)

if new_hash == stored_hash:
    login_success()
```

The extraction is done automatically by `checkpw()`.

---

## Why is this useful?

Imagine two users have the same password:

```text
User A -> secret123
User B -> secret123
```

Without salt:

```text
hash(secret123) = abc123
hash(secret123) = abc123
```

An attacker immediately knows both users share the same password.

With bcrypt salts:

```text
User A -> $2b$12$saltA...
User B -> $2b$12$saltB...
```

Hashes become:

```text
$2b$12$saltA...hashA
$2b$12$saltB...hashB
```

They look completely different, even though the password is identical.

That's why bcrypt generates a unique salt for every password and embeds that salt directly into the stored hash.

Yes, generally you need a **separate API endpoint** for generating a new access token using a refresh token.

The common JWT authentication flow looks like this:

```
Login
  |
  | (email + password)
  ↓
Backend
  |
  | creates:
  | - access_token (short expiry: 15 min)
  | - refresh_token (long expiry: 7 days / 30 days)
  ↓
Frontend stores both tokens
```

---

## 1. User Login

Frontend sends:

```http
POST /login
```

Body:

```json
{
  "email": "user@test.com",
  "password": "123456"
}
```

Backend verifies password and returns:

```json
{
  "access_token": "eyJhbGciOiJIUzI1...",
  "refresh_token": "eyJhbGciOiJIUzI1...",
  "token_type": "bearer"
}
```

Example expiry:

```
access_token  -> 15 minutes
refresh_token -> 7 days
```

---

## 2. Frontend uses access token

Every API request:

```http
GET /profile

Authorization: Bearer access_token
```

Backend checks:

```python
jwt.decode(access_token)
```

If valid:

```
allow request
```

If expired:

```
return 401 Unauthorized
```

Example:

```json
{
    "detail": "Token expired"
}
```

---

## 3. Frontend detects expired token

Frontend receives:

```
HTTP 401
```

Then it calls refresh API:

```http
POST /refresh-token
```

with refresh token:

```json
{
    "refresh_token": "eyJhbGciOiJIUzI1..."
}
```

---

## 4. Backend verifies refresh token

Backend:

```python
def refresh_access_token(refresh_token):

    payload = jwt.decode(
        refresh_token,
        SECRET_KEY,
        algorithms=["HS256"]
    )

    user_id = payload["user_id"]

    new_access_token = create_access_token({
        "user_id": user_id
    })

    return new_access_token
```

Response:

```json
{
    "access_token": "new_access_token"
}
```

---

## 5. Frontend retries the original request

Before:

```
GET /profile
Authorization: expired_access_token
```

After refresh:

```
GET /profile
Authorization: new_access_token
```

User doesn't notice anything.

---

## FastAPI example

### Login API

```python
@app.post("/login")
def login():

    access_token = create_access_token(
        {"user_id": 1},
        expires_minutes=15
    )

    refresh_token = create_refresh_token(
        {"user_id": 1},
        expires_days=7
    )

    return {
        "access_token": access_token,
        "refresh_token": refresh_token
    }
```

---

### Refresh API

```python
@app.post("/refresh-token")
def refresh_token(refresh_token: str):

    payload = verify_refresh_token(refresh_token)

    if not payload:
        raise HTTPException(
            status_code=401,
            detail="Invalid refresh token"
        )

    new_access_token = create_access_token({
        "user_id": payload["user_id"]
    })

    return {
        "access_token": new_access_token
    }
```

---

## Where store refresh token?

Common approaches:

### Option 1: Database (recommended)

Table:

```
user_tokens

id
user_id
refresh_token
expires_at
created_at
```

Example:

```
1 | 10 | abcxyz123 | 2026-06-20
```

Advantages:

* Can revoke tokens
* Logout from all devices
* Track sessions

---

### Option 2: Stateless JWT refresh token

Just create another JWT:

```
Access Token:
{
 user_id:1,
 exp: 15 minutes
}


Refresh Token:
{
 user_id:1,
 exp: 7 days,
 type:"refresh"
}
```

No database needed.

---

## Important security practices

### 1. Do not make access tokens long-lived

Bad:

```
access_token = 30 days
```

If stolen, attacker has access for 30 days.

Better:

```
access_token = 10-15 minutes
refresh_token = 7-30 days
```

---

### 2. Store refresh token securely

For web apps:

```
Refresh token → HttpOnly Secure Cookie
Access token  → Memory
```

Avoid:

```
localStorage
```

for sensitive tokens because JavaScript can access it.

---

### 3. Rotate refresh tokens

A more secure flow:

```
refresh token used
        |
        ↓
invalidate old refresh token
        |
        ↓
generate new refresh token
```

---

So the answer is: **Yes, normally you create a `/refresh-token` API. The frontend calls it whenever the access token expires, sends the refresh token, and receives a new access token.**

Q :

frontend where should be store refresh and access token in browser side ?

Where to store JWT tokens in the browser depends on the type of application and security requirements. The common recommendation is:

| Token             | Recommended storage                | Reason                                                      |
| ----------------- | ---------------------------------- | ----------------------------------------------------------- |
| **Access token**  | Memory (JavaScript variable/state) | Short-lived, less exposed to attacks                        |
| **Refresh token** | `HttpOnly + Secure Cookie`         | JavaScript cannot read it, protects against XSS token theft |

---

## Recommended flow for a web application

### 1. Login response

Backend returns:

```json
{
  "access_token": "eyJhbGciOi...",
  "expires_in": 900
}
```

And sets refresh token as a cookie:

```http
Set-Cookie:
refresh_token=abcxyz;
HttpOnly;
Secure;
SameSite=Strict;
Max-Age=604800
```

The browser automatically stores the refresh token cookie.

---

### 2. Frontend stores access token in memory

Example React:

```javascript
let accessToken = null;

function setToken(token) {
    accessToken = token;
}
```

When calling APIs:

```javascript
fetch("/profile", {
    headers: {
        Authorization: `Bearer ${accessToken}`
    }
});
```

---

### 3. Access token expires

API response:

```http
401 Unauthorized
```

Frontend calls:

```http
POST /refresh-token
```

It does **not** send the refresh token manually.

The browser automatically sends:

```http
Cookie:
refresh_token=abcxyz
```

Backend verifies it and returns:

```json
{
    "access_token": "new_access_token"
}
```

Frontend replaces the old access token in memory.

---

## What about localStorage?

Many applications do this:

```javascript
localStorage.setItem(
    "access_token",
    token
);
```

It is easy, but there is a security risk.

If your site has an XSS vulnerability:

```javascript
// attacker script
localStorage.getItem("access_token")
```

The attacker can steal the token.

---

## What about sessionStorage?

```javascript
sessionStorage.setItem(
    "access_token",
    token
);
```

It is slightly safer because it disappears when the tab closes, but JavaScript can still access it, so XSS can still steal it.

---

## Common production setup

```
Browser
│
├── Memory
│     └── Access Token
│          └── expires in 10-15 minutes
│
└── HttpOnly Secure Cookie
      └── Refresh Token
           └── expires in 7-30 days
```

---

## For mobile apps

The approach is different:

* Android → Keystore
* iOS → Keychain

because there is no browser cookie environment.

---

For a FastAPI + React application, a good architecture is:

```
React
 |
 | Authorization: Bearer access_token
 ↓
FastAPI
 |
 | refresh_token cookie
 ↓
/refresh-token endpoint
```

This gives you short-lived access tokens and a safer long-lived session mechanism.

Yes, exactly. If the access token is stored **only in memory**, it will be removed when the user:

* closes the browser tab
* refreshes the page
* closes the browser completely

So how does the user stay logged in?

The answer is: **the refresh token cookie**.

### Example flow

#### First login

User logs in:

```
Frontend
   |
   | email + password
   ↓
FastAPI
   |
   | creates:
   | - access_token (15 min)
   | - refresh_token (7 days)
   ↓
Browser
   |
   | access_token → memory
   | refresh_token → HttpOnly Cookie
```

---

### User closes browser

Memory is cleared:

```
access_token ❌ removed
```

But cookie remains:

```
refresh_token ✅ still exists
```

because the cookie has an expiry time:

```
refresh_token
expires: 7 days
HttpOnly: true
Secure: true
```

---

### User opens website again

Frontend starts:

```
React App loaded
```

It checks:

```
Do I have access_token?
```

Result:

```
No
```

Then frontend calls:

```
POST /refresh-token
```

The browser automatically sends:

```
Cookie:
refresh_token=abc123
```

FastAPI:

1. Reads refresh token.
2. Verifies it.
3. Creates a new access token.

Response:

```json
{
  "access_token": "new_access_token"
}
```

Frontend puts it back into memory:

```javascript
let accessToken = "new_access_token";
```

Now the user is logged in again.

---

### Real-world flow

```
First login
     |
     ↓
Access Token (memory)
Refresh Token (cookie)
     |
     ↓
Close browser
     |
     ↓
Memory cleared
Cookie remains
     |
     ↓
Open browser again
     |
     ↓
Call /refresh-token
     |
     ↓
Get new access token
     |
     ↓
Continue using app
```

---

### What if refresh token also expires?

Example:

```
Access token: 15 minutes
Refresh token: 30 days
```

After 30 days:

```
/refresh-token
        |
        ↓
401 Unauthorized
```

Frontend redirects user to:

```
/login
```

and the user must authenticate again.

---

So the reason we keep the **refresh token in an HttpOnly cookie** is exactly for this situation: the user can close and reopen the browser while keeping their session without storing the access token permanentl

JWT working meachnism =>

how jwt verify that tocken is not modified and apyload not modified ?

Great question. The security of JWT comes from its **signature**.

When you create a JWT:

```python
payload = {
    "sub": "123",
    "role": "admin"
}

token = jwt.encode(
    payload,
    SECRET_KEY,
    algorithm="HS256"
)
```

JWT creates three parts:

```text
header.payload.signature
```

For example:

```text
eyJhbGciOiJIUzI1NiIs...
.
eyJzdWIiOiIxMjMiLCJyb2xlIjoiYWRtaW4ifQ...
.
abc123xyz...
```

### How the Signature is Created

With HS256, the signature is roughly:

```text
HMAC_SHA256(
    base64(header) + "." + base64(payload),
    SECRET_KEY
)
```

Suppose:

```python
payload = {
    "sub": "123",
    "role": "admin"
}
```

The server generates:

```text
header.payload.signature
```

using:

```python
SECRET_KEY = "my-secret-key"
```

---

### What Happens If Someone Modifies the Payload?

Imagine an attacker decodes the token and changes:

```json
{
    "sub": "123",
    "role": "admin"
}
```

to

```json
{
    "sub": "123",
    "role": "superadmin"
}
```

Now the token becomes:

```text
header.modified_payload.original_signature
```

When FastAPI verifies the token:

```python
jwt.decode(
    token,
    SECRET_KEY,
    algorithms=["HS256"]
)
```

PyJWT:

1. Reads header and payload.
2. Recomputes the signature using your `SECRET_KEY`.
3. Compares the recomputed signature with the signature inside the token.

```text
Expected Signature  = xyz789
Token Signature     = abc123

Not equal ❌
```

Result:

```python
jwt.exceptions.InvalidSignatureError
```

The token is rejected.

---

### Why Can't the Attacker Generate a New Signature?

Because they don't know:

```python
SECRET_KEY
```

Without the secret key, they cannot create a valid signature.

Think of it like this:

```text
Payload + Secret Key
        ↓
   Signature
```

Anyone can see the payload, but only the server knows the secret key needed to create a valid signature.

---

### Verification Flow

```text
Client sends JWT
        ↓
Server decodes header/payload
        ↓
Server uses SECRET_KEY
        ↓
Recreates signature
        ↓
Compare signatures
        ↓
Match? → Valid
No Match? → Reject
```

---

### Example

Original token:

```json
{
    "sub": "1",
    "role": "user"
}
```

Attacker changes it to:

```json
{
    "sub": "1",
    "role": "admin"
}
```

The payload changed, so the signature should also change.

Since the attacker doesn't know your secret key, they cannot generate the correct new signature.

FastAPI/PyJWT detects the mismatch and raises:

```python
InvalidSignatureError
```

---

### One More Important Point

JWT **protects against modification**, but **does not hide data**.

Anyone can decode:

```json
{
    "sub": "1",
    "role": "admin"
}
```

and read it.

JWT guarantees:

* ✅ Payload was not modified.
* ✅ Token was issued by someone who knows the secret key.
* ✅ Expiration (`exp`) can be enforced.
* ❌ Payload is not encrypted.
* ❌ Payload is not secret.

That's why the common practice is to store only things like:

```python
{
    "sub": "123",
    "role": "user",
    "exp": ...
}
```

and never passwords, OTPs, API keys, or other sensitive secrets.
